## CRITICAL — 즉시 수정 필요

### 1. ROI 삭제 후 인덱스 범위 초과
**파일:** `ui.js` — `_unlinkCodeRoi()`, `deleteRoiFromPopup()`

**문제:**
코드박스 여러 개 삭제 후 답안 ROI 인덱스를 재계산하는데, 삭제 결과 `newIdx`가 배열 길이 이상이 될 수 있음.
```javascript
const deleted = codeIndices.filter(ci => ci < answerRoiIdx).length;
const newIdx = answerRoiIdx - deleted;
// newIdx가 imgObj.rois.length 이상이면 → undefined 접근
```

**수정 방향:**
`newIdx`에 `Math.min(newIdx, imgObj.rois.length - 1)` 바운드 체크 추가.

**기존 로직 충돌:** 없음. 바운드 체크만 추가하는 것이라 다른 코드에 영향 없음.

### 2. 업로드 병렬화 race condition
**파일:** `imageManager.js` — `handleUpload()`

**문제:**
4스트림 병렬 처리에서 `completed` 카운터가 정확히 `total`에 도달하면 `onAllDone()` 호출. 하지만 `processImage` 콜백이 비동기(toBlob)라서 2개의 스트림이 거의 동시에 완료하면 `completed >= total` 체크가 2번 true가 될 수 있음 → `onAllDone()` 중복 호출.

**수정 방향:**
`onAllDone` 호출부에 1회 실행 가드 추가:
```javascript
let finished = false;
const onAllDone = () => {
    if (finished) return;
    finished = true;
    // ... 기존 로직
};
```

**기존 로직 충돌:** 없음. 가드 플래그만 추가. 기존 로직 변경 없음.

### 3. 세션 저장 중 동시 저장 방지
**파일:** `session.js` — `saveCurrentSession()`

**문제:**
세션 저장이 async인데 락이 없음. 자동 저장 + 수동 저장이 동시에 호출되면 같은 파일에 동시 쓰기 → 파일 깨짐 가능.

**수정 방향:**

```javascript
if (this._saveLocked) return;
this._saveLocked = true;
try { /* 기존 저장 로직 */ } finally { this._saveLocked = false; }
```

**기존 로직 충돌:** 없음. try-finally로 감싸기만 하면 됨. 저장 로직 자체 변경 없음.

## HIGH — 안정성 문제

### 4. ensureLoaded 타임아웃 없음
**파일:** `imageManager.js` — `ensureLoaded()`

**문제:**
이미지 파일이 손상/삭제되었으면 `onload`도 `onerror`도 안 불릴 수 있음 → Promise가 영원히 안 끝남 → 일괄처리/세션저장이 멈춤.

**수정 방향:**
```javascript
ensureLoaded(imgObj) {
    return new Promise((resolve) => {
        // ... 기존 로직 ...
        // 타임아웃 10초
        setTimeout(() => {
            if (!resolved) { resolved = true; resolve(null); }
        }, 10000);
    });
}
```

**기존 로직 충돌:** 없음. 타임아웃 추가만. 기존 콜백 로직 변경 없음. 타임아웃 시 null 반환 → 호출부에서 이미 null 체크하고 있음.

### 5. linkedCodeRoiIds null 체크
**파일:** `scoring.js`, `grading.js`

**문제:**
```javascript
const _codeIds = roi.settings.linkedCodeRoiIds || (roi.settings.linkedCodeRoiId ? [...] : []);
```
`linkedCodeRoiIds`가 명시적 `null`이면 falsy → 오른쪽으로 감. 하지만 만약 빈 배열 `[]`이면 truthy → for-of 실행 → 0회 루프 → 문제 없음.
실제 위험: `linkedCodeRoiIds`가 `null`이 아닌 잘못된 타입(숫자, 문자열)이면 `for...of`에서 TypeError.

**수정 방향:**
```javascript
const _codeIds = Array.isArray(roi.settings.linkedCodeRoiIds) 
    ? roi.settings.linkedCodeRoiIds 
    : (roi.settings.linkedCodeRoiId ? [roi.settings.linkedCodeRoiId] : []);
```

**기존 로직 충돌:** 없음. 타입 체크만 강화. 정상 데이터에서 동작 동일.

### 6. 이벤트 리스너 누적
**파일:** `session.js` — `init()`

**문제:**
`init()`에서 버튼에 addEventListener를 등록하는데, 세션 전환 시 `init()`이 다시 호출되지는 않으므로 **현재는 실제 문제 아님**. 하지만 `onBeforeClose` 리스너는 1회 등록이 보장되지 않을 수 있음.

**수정 방향:**
`init()` 내 리스너 등록부에 중복 방지 플래그:
```javascript
if (this._initDone) return;
this._initDone = true;
```

**기존 로직 충돌:** 없음. init()은 앱 시작 시 1회 호출이므로 현재 실무적으로 문제 없음. 방어적 코드.

## 18. 수험번호 블롭 불안정 — 진하기 차이로 해결 가능
**파일:** `batch.js` — `_retryFailedImages()`, `omrEngine.js`

**문제 분석 (수험번호블롭확인.txt):**

**진하기 400 (기본):**
```
행=10 열=7 (67블롭)
열평균=[34,95,155,216,262,323,384,445,505,581]
  → 이상=2: 4→5번 간격 46px, 9→10번 간격 76px (기대 61px와 불일치)
Q1: gap=0.020 → null  (1등=0.253, 2등=0.233 — 차이 거의 없음)
Q3: gap=0.018 → null
Q4: gap=0.001 → null
Q5: gap=0.022 → null
```

**진하기 500 (+100%):**
```
행=10 열=7 (71블롭)
열평균=[34,94,155,216,277,338,399,460,520,581]
  → 이상=0: 모든 간격 60~61px (정상!)
Q1: gap=0.619 → 8 (1등=1.000)
Q3: gap=0.620 → 8
Q4: gap=0.647 → 8
Q5: gap=0.668 → 8
```

**근본 원인:**
진하기 400에서는 5~10번 열 위치의 블롭이 불안정하게 감지됨 (cy=262,323,384,445,505,581 — 간격 불균일). 이 때문에 sampleCell이 엉뚱한 위치를 읽어서 comp가 전체적으로 0.2~0.4로 비슷 → gap 부족 → null.

진하기 500에서는 블롭이 정확한 위치에 잡힘 (cy=277,338,399,460,520,581 — 간격 균일) → sampleCell이 정확한 위치 → 마킹된 버블 comp=1.000.

**이건 이미 retry 로직에서 처리됨.** retry의 시도1(+100%)에서 진하기 500으로 재분석하면 해결. 다만 retry 대상 판별에서 이 이미지가 잡히는지 확인 필요 — Q1,Q3,Q4,Q5가 null이고 maxFill이 0.25~0.43이라 `maxFill >= 0.35` 조건에 걸림 → 인식 오류로 판정 → retry 대상 O.

**추가 수정 불필요.** 기존 retry 로직으로 커버됨.
