# 데이터 취합 / 마스터 DB 설계 (2026-04-27)

> **상태**: 설계 완료, 구현 미시작
> **우선순위**: 최상 (Phase 1 시급)
> **작성 배경**: 시험을 일/주/월별로 빈번하게 진행하고, 같은 학생이 다양한 시험에 응시하는 환경에서, 시간이 흐른 뒤 오류 없이 학생 개인 데이터를 취합하기 위한 데이터 구조 설계.

## 요구사항 정리

| 항목 | 내용 |
|---|---|
| 시험 주기 | 일별 / 주별 / 월별 (혼재) |

## 현재 세션이 저장 중인 데이터 (변경 전)

`session.js`가 저장하는 한 세션 = 한 시험:
- `students[]` — 시험인원: `{examNo, name, birth, phone}`
- `subjects[]` — 과목 메타: `{name, code, answers, scorePerQuestion, scoreMap…}`
- `answerKey` — 전역 정답키 (다과목 시 보통 미사용)
- `subjectMerges[]` — A/B 과목 합치기
- `periods[]` — 다교시 구성
- `images[]` — 각 학생 OMR 분석/채점 결과

## 깨지기 쉬운 지점 — 미래 취합 시 위험

### ① 학생 식별
- `examNo`는 시험마다 새로 매겨짐 → 시험 간 매칭 불가
- 같은 학생이 시험 1: "이민범", 시험 2: "이 민범" (공백 차이) → 이름만으론 매칭 실패
- 전화번호: 가장 안정적이지만 미기입/오기 가능
- 생년월일 형식 불일치 (`040124` vs `2004-01-24`) 가능

### ② 과목명 표준화
- "수학", "수학(A)", "수학A"가 같은 과목인지 구분 불가
- `SubjectManager.findByRoiName` 매핑이 세션 안에만 존재 → 시험 간 매핑 규칙 다르면 통계 망가짐

### ③ 스키마 진화
- 1년 뒤 새 필드 추가/제거 시 옛 세션과 호환 안 될 수 있음
- 현재 `version: 1`만 있고 마이그레이션 로직 없음

## 권장 아키텍처

### 폴더 구조

```
앱 데이터 폴더/
├── master/                       ← 시간이 흘러도 변하지 않는 핵심
│   ├── students.json             ← 마스터 학생 DB (영구 ID)
│   ├── subjects.json             ← 표준 과목명 사전
│   └── exam_groups.json          ← 시험 그룹 정의
├── sessions/                     ← 기존 그대로 (편집 중인 작업 파일)
│   └── *.json
└── exam_results/                 ← 채점 완료된 시험 결과 (취합 대상)
    └── 2026-04-25_LEET-2회.json
```

### 학생 마스터 (master/students.json)

```json
[
  {
    "studentId": "stu_a3f9d2",              // 내부 영구 ID (UUID v4 또는 short ID)
    "displayId": "2024-001",                // 사람이 외우는 학번 (선택)
    "name": "이민범",
    "birthYmd": "20040124",                  // 8자리 형식 통일
    "phone": "01012345678",                  // 하이픈 제거 정형화
    "school": "...",
    "grade": "고3",
    "createdAt": "2026-04-25",
    "updatedAt": "2026-04-25"
  }
]
```

**매칭 우선순위 (중복 등록 방지)**:
1. `phone` 일치 → 같은 학생
2. `birthYmd + name` 일치 → 같은 학생
3. 둘 다 안 맞으면 신규 추가 (UUID 자동 발급)

### 시험 결과 (exam_results/{examId}.json)

```json
{
  "schemaVersion": 2,
  "examId": "2026-04-25_LEET-2회",
  "examName": "LEET 2회 전국모의고사",
  "examDate": "2026-04-25",
  "examGroup": "LEET 월간 모의고사 2026",   // 그룹 추이용 (선택)
  "tags": ["LEET", "추리논증"],
  "subjects": ["추리논증"],
  "students": [
    {
      "studentId": "stu_a3f9d2",            // 마스터 ID (필수)
      "examNo": "7001700",                   // 이 시험에서만 유효한 임시 번호
      "displayName": "이민범",                // 표시용 (출력 시점 스냅샷)
      "subjects": {
        "추리논증": {
          "score": 25,
          "rank": 417,
          "percentile": 38.0,
          "correctCount": 25,
          "totalPossible": 40,
          "answers": [...]
        }
      },
      "totalScore": 25,
      "totalRank": 417,
      "_noOmr": false
    }
  ]
}
```

핵심: **학생당 `studentId`만 일관되게 부여되면 모든 취합이 가능**.

### 표준 과목 (master/subjects.json)

```json
[
  {
    "code": "ETHICS",
    "standardName": "추리논증",
    "aliases": ["추리논증", "추론", "Reasoning"],
    "createdAt": "2026-04-25"
  }
]
```

ROI 이름 → standardName 매핑은 세션 외부 전역에서 관리.

### 시험 그룹 (master/exam_groups.json)

```json
[
  {
    "groupId": "leet_monthly_2026",
    "name": "LEET 월간 모의고사 2026",
    "description": "2026년 월간 시리즈",
    "tags": ["LEET", "월간"],
    "createdAt": "2026-01-01"
  }
]
```

## 취합 자동화 — 어떻게 동작하는가

### (a) 학생 개인 추이
```
exam_results/*.json 전부 읽기
→ studentId == "stu_a3f9d2"인 항목만 필터
→ examDate 순 정렬
→ 시험별 점수/석차 시계열
```

### (b) 시험 그룹별 추이
```
examGroup == "LEET 월간 모의고사 2026"인 시험들 필터
→ 각 시험의 평균/표준편차/응시인원 비교 (회차별 그래프)
```

### (c) 한 시험 안에서의 추이
- 현재 기능 그대로 (이미 잘 됨)

이 셋이 **같은 데이터 소스**(exam_results/)에서 다 나옴.

## 단계별 도입 로드맵

### Phase 1 — 학생 마스터 DB 도입 ⭐ 1순위
- [ ] `master/students.json` 자동 생성 + 빈 스키마
- [ ] 시험인원 등록 시 마스터 검색·매칭 UI 추가 (전화/이름+생일 우선순위)
- [ ] 매칭 못한 학생 자동 마스터 추가 (UUID 발급)
- [ ] 기존 session 저장 시 학생 객체에 `studentId` 추가
- [ ] 전화·생년월일 자동 정형화 (하이픈 제거, 8자리 형식)
- [ ] 기존 데이터 마이그레이션 1회성 스크립트

→ **이거 하나만 해놔도** 미래 어느 시점에든 취합 가능.

### Phase 2 — 시험 메타 + 그룹
- [ ] 세션 생성 시 `examGroup`, `tags` 입력 옵션
- [ ] 시험 그룹 관리 모달
- [ ] `master/exam_groups.json` 관리

### Phase 3 — 표준 export
- [ ] "교정확정" 시점에 `exam_results/{examId}.json` 자동 export
- [ ] 형식: schemaVersion 2 (위 스키마)
- [ ] export 파일은 **읽기 전용** (취합 안정성)

### Phase 4 — 누적 분석 모드
- [ ] 새 탭 또는 별도 페이지: "성적 누적 분석"
- [ ] 학생 검색 → 그 학생의 시험별 추이 그래프
- [ ] 시험 그룹 선택 → 회차별 통계 비교
- [ ] 취합 결과 export (CSV / 엑셀)

### Phase 5 (옵션) — SQLite로 이전
- [ ] 시험·학생이 많아져 JSON 로드 느려지면 SQLite로 마이그레이션
- 지금은 JSON으로 충분 (수천 시험 / 수만 학생까지는 부담 없음)

## 다른 PC 동기화 — 학원 서버 없이

학원 서버는 못 쓰지만 **클라우드 동기화 폴더**(Dropbox / OneDrive / Google Drive)를 활용:
- `master/`와 `exam_results/`를 동기화 폴더 안에 두면 자동 동기화
- 집PC ↔ 학원PC 시간차 작업 OK
- 동시 편집은 충돌 가능 → 한 번에 한 PC만 사용 권장
- 비용 0 / 인프라 0

설정 항목 추가 필요:
- "마스터 DB 폴더 경로" — 사용자가 직접 지정 (예: `C:\Users\admin\Dropbox\OMR\master`)

## 결정 필요 사항 (구현 착수 전)

### Q1. Phase 1을 언제 시작할까?
- 기존 데이터 마이그레이션은 1회성 스크립트로 처리
- 시험인원 등록 UI에 "마스터에서 검색" 기능 추가
- 점진적 도입 가능 (백그라운드에서 ID만 부여)

### Q2. 학생 표시용 ID는 어떻게?
- (a) 자동 UUID만: 단순하지만 사람이 못 외움
- (b) 학원 학번 입력 필수
- (c) **둘 다 (권장)**: UUID는 시스템용, 학번은 표시용

### Q3. 클라우드 동기화 폴더 사용?
- 동기화 폴더 위치를 설정 항목으로 추가 → 마스터 DB를 거기에 두면 끝
