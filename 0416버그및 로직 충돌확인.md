## 1. 좌표계 혼용 버그

### 1-1. Stage4-4 누락행 복원 좌표계 (✅ 방금 수정)
- **증상:** `backMargin=-138` 같은 음수 발생 → 앞쪽에만 계속 행 추가
- **원인:** `blob.cx/cy`는 **절대 좌표** (offset 포함), `roiDim`은 **ROI 내부 크기**
- **수정:** `roiTop = offsetY`, `roiBottom = offsetY + height`로 절대 좌표 통일

### 1-2. Stage3 열 평균 — 상대/절대 혼재 가능성
- **코드:** `_stage3_filterByColumnConsistency` — blob의 cx/cy 평균을 colAvgX로 사용
- **의심:** blob 좌표가 일부는 상대(Stage1 원본), 일부는 절대(Stage4에서 offset 더한 후)일 수 있음
- **확인 필요:** 해당 함수 호출 시점에 blob 좌표계가 무엇인지 추적

### 1-3. `_resampleRow`의 blob.cx/cy
- **문제:** `const cx = blob.cx - offsetX` — 절대 좌표라고 가정하고 offset 빼는데, 실제로 그 시점에 절대인지 보장 안 됨

## 2. 양식(Template) blobPattern 관련

### 2-1. 절대 vs 비율 혼재
- **저장:** `bubbleW`(절대) + `bubbleWRatio`(비율) 둘 다
- **사용:** `analyzeROI`에서 절대값 우선, 없으면 비율 × ROI 크기
- **문제:** 두 값이 서로 모순될 수 있음. 스캔 해상도 다르면 absolute 값 부정확
- **현재 상태:** Stage4는 양식 값 무시하고 감지 중앙값 사용 (사용자 요구 반영됨)

### 2-2. `rowYRatios`/`colXRatios`는 저장되지만 사용 안 함
- **현재:** 저장은 하지만 Stage4-4에서 사용 안 함 (감지값만 씀)
- **용도:** 향후 "어느 순번이 비었는지" 매칭 구현 시 필요
- **결정 필요:** 완전히 제거할지, 보존할지

### 2-3. 구버전 양식 호환성
- **상황:** 사용자가 예전에 저장한 양식은 `rowYRatios`, 절대 픽셀값 없음
- **현재:** 비율값은 있으니 일부 fallback 동작, 하지만 Stage4는 양식 무시
- **문제 없음**, 단 사용자에게 "양식 재저장 권장" 안내 필요

## 3. Stage4-4 누락 행 복원 — 잠재 이슈

### 3-1. 앞/뒤 extrapolation 휴리스틱 한계
- **현재:** `frontMargin`, `backMargin` 비교로 어느 쪽이 비었는지 추정
- **한계:** 사용자가 ROI를 비대칭으로 그리면 오판 가능
- **예:** 실제로는 마지막 행 누락인데 앞쪽이 유난히 타이트하면 앞에 추가해버림
- **개선 아이디어:** 양식의 `rowYRatios`로 정확히 어느 순번 누락인지 매칭

### 3-2. 중간 gap 판정 민감도
- **현재:** `Math.round(gap / detectedGap) - 1`
- **경계 케이스:** gap이 detectedGap × 1.4 수준이면 round = 1 → 1개 누락 판정. 애매함.
- **개선:** 1.5배 이상일 때만 누락으로 판정

### 3-3. 감지 행이 1개뿐인 경우
- **현재:** `detectedGap` 계산 불가 → 스킵
- **문제:** 그러면 복원 로직 전혀 작동 안 함
- **개선:** 양식에 저장된 절대 gap 사용해서라도 복원 시도

## 4. 1배 판정 / 1.5배 링 교차 검증

### 4-1. 임계값 점검 필요
- **현재 (omrEngine.js line ~724):**
  - 동그란: `gap≥0.12 OR f≥0.95`
  - 세로길쭉: `gap≥0.12 OR f≥0.95 AND e≥0.25`
- **의심:** f≥0.95가 너무 엄격 → 마킹이 약하면 null로 빠짐
- **결정 필요:** 실제 데이터로 튜닝

### 4-2. `_xvMatch` 플래그와 row 상태 불일치
- **현재:** conflict, bubble_only, xv_only, match, both_null
- **문제:** setAnswerAndAdvance 호출 후 `_xvAutoCorrected = false`로 변경하는데, `_xvMatch`는 그대로
- **영향:** 교정 탭에서 이력으로 이동은 되지만, 내부 플래그 불일치
- **위험 낮음**, 정리만 필요

### 4-3. 링 4분면 계산 시 버블 경계선 포함
- **현상:** 버블 테두리선이 링 영역에 걸쳐 ringFill 이 과대 평가됨
- **현재 대응:** `ringMax` MAX 임계 0.7 + gap 0.3 필요 (보수적)
- **개선:** 링 안쪽 1px 제외 로직 추가 가능

## 5. 교정 탭 (correction.js)

### 5-1. `row._correctionInitial` 결정 시점
- **현재:** collect() 첫 호출 시 고정
- **문제:** 사용자가 분석 탭에서 교정한 걸 교정 탭 진입 전에 했으면 초기 상태로 기록됨
- **경미한 UX 이슈**

### 5-2. setAnswerAndAdvance의 Grading.grade 호출
- **현재:** 교정할 때마다 전체 이미지 재채점
- **성능:** 많은 이미지 × 많은 교정 시 느려질 수 있음
- **개선:** 해당 문항만 갱신하거나, 디바운스

### 5-3. 이력 칼럼 팝업의 썸네일 클릭 핸들러
- **구현:** 팝업에서 원본 핸들러를 override 하는데, setAnswer 호출 후 overlay.remove
- **잠재 이슈:** render() 호출로 DOM이 재생성되면서 overlay 참조가 무효화될 가능성
- **현재 동작:** 정상 작동하지만 리팩토링 여지
